## 摘要

数据清洗是一项繁琐的数据准备任务，通常需要用户以预定义配置的形式进行监督，例如规则、参数或模式。我们最近开发了两个无配置系统 Raha 和 Baran，用于以半监督方式检测和纠正数据错误。本文展示了这两个系统如何用于端到端的数据清洗流水线。

我们的演示表明，通过有效的特征表示、标签传播和迁移学习方法，用户监督可以减少到几乎可以忽略不计的少量示例纠正。虽然检测和纠正这两个清洗步骤面临的挑战有显著差异，但我们基于同一个直觉设计了相应系统。两个系统内部都利用一组可自动生成的基础检测器和纠错器，并通过少量用户标签学习如何组合它们。

在实践中，只需少量用户标注的 20 个元组，就可以有效识别并修复数据集中的数据质量问题。此外，两个系统都能从先前清洗任务的知识中受益。通过迁移学习，两个系统可以分别在错误检测运行时间和错误纠正效果方面优化当前数据清洗任务。

## 1. 引言

数据质量管理一直是最难自动化的任务之一。事实上，数据清洗是数据科学家最重要但也最耗时的任务之一 [7]。数据清洗任务包含两个主要步骤：（1）错误检测；（2）错误纠正。错误检测的目标是识别错误或脏的数据值 [1]。错误纠正的目标是修复这些错误值 [19]。

围绕端到端数据清洗及其若干子问题，已有大量研究。大多数传统数据清洗系统遵循预配置范式，即要求用户提供正确且完整的一组规则 [8, 3]、参数 [23]，或二者兼有 [19]。例如，数据清洗系统可能需要数据模式，例如日期格式 `dd.mm.yyyy`，也可能需要统计参数，例如期望均值和标准差。然而，提前提供正确且完整的规则和参数集合，对大多数非专家用户来说是一个主要障碍，因为他们需要非常了解数据集和数据清洗系统，才能正确配置系统 [1, 15, 23]。

最近，我们提出了用于数据清洗的无配置范式，该范式更适合领域专家用户，而这些用户并不擅长为复杂数据清洗系统生成配置。具体来说，我们提出了错误检测系统 Raha [15]，以及用于错误纠正的配套系统 Baran [14]。两个系统遵循同一个统一思想：它们在内部以半监督方式聚合一组基础错误检测器或纠错器。用户只需参与标注被选中的数据点，即标记或修复少量数据错误。换言之，用户不需要提供任何依赖数据或依赖工具的配置。

两个系统都学习如何把用户提供的错误检测或纠错示例推广到数据集的其余部分。为此，每个系统都需要不同类型的特征构造方式，用于描述并暴露错误值。由于数据错误通常依赖于数据集和使用场景，并非每个基础检测器都对所有可能的数据集有用。尤其是检测器的参数配置，例如简单离群点检测技术中的数值参数，依赖于当前数据集的特征。

与其他要求用户配置基础检测器的错误检测聚合器 [1, 21] 不同，Raha 的设计并不关心检测器本身质量如何，只要有足够大且足够多样的一组检测器可用即可。Raha 转而使用这些检测器的输出来对值进行聚类。借助用户示例，Raha 随后识别脏值聚类，并传播用户标签以提升分类准确率。

另一方面，Baran 需要一个特征向量来编码针对已检测数据错误的大量可能纠正候选。在理论上，任意可能的字符串都可能成为某个数据错误的纠正值。通过利用最容易获得的错误上下文，即值本身、值所在的领域以及相邻列中的相关值，Baran 生成一大组潜在纠正，然后通过人工监督对其进行有效过滤。

此外，Raha 和 Baran 的设计都能从迁移学习中受益 [17]。事实上，Raha 和 Baran 都可以从先前已清洗的数据集中学习，以提升当前数据集上的数据清洗性能，并减少用户参与。Raha 会计算当前数据集与先前已清洗数据集之间的相似度，从而为当前数据集剪枝不相关的错误检测器。通过这种方式，Raha 可以降低整体数据清洗运行时间，同时不损害整体检测过程的有效性。

Baran 从任何已清洗数据集中抽取基于值的纠正，用于预训练基础纠错器。这些预训练纠错器之后可以在当前数据集上进行微调。预训练不仅让 Baran 无需用户标注即可获得高频主题纠正，例如把 `US` 修正为 `United States`，还会生成更多纠错证据，从而提高底层学习算法的收敛速度。

在先前论文中，我们解释了 Raha [15] 和 Baran [14] 的理论方面；而在本文中，我们关注两个系统各自的技术方面，以及串联 Raha 与 Baran 的端到端数据清洗流水线。具体而言，我们演示以下方面：

- 展示 Raha 和 Baran 如何把少量用户标注的错误或纠错示例推广到数据集其余部分。我们的演示支持深入查看相似的数据错误组、这些组如何通过大量错误检测器聚类，以及建议纠正如何生成。
- 展示 Raha 和 Baran 如何使用先前已清洗数据集和迁移学习来提升数据清洗性能，并减少数据清洗过程中的用户参与。
- 展示为特定系统设计的不同采样技术如何影响端到端流水线性能。

我们的 Raha 和 Baran 系统均已在线发布。代码仓库还包含作为用户界面的交互式 Jupyter Notebook。此外，我们也将系统打包并上传到了 Python Package Index，也就是 PyPI。我们的 Python 包可以通过 `pip` 包管理器轻松安装和管理。

### 2.1 错误检测引擎

脏数据集首先由 Raha 处理以检测其中的数据错误。在内部，Raha 会生成并运行大量基础错误检测器 `S = {s1, s2, ..., s|S|}`。这些检测器代表了传统错误检测技术的四个主要类别 [1]。Raha 采用系统化方法，针对以下错误检测算法生成多种配置，并将它们作为基础检测器。

1. 离群点检测算法 [18] 从数据值与对应列中值的总体分布是否兼容的角度评估数据值的正确性。我们利用直方图建模和高斯建模 [18] 作为两种基础离群点检测算法，二者分别利用数据值的出现情况和数值大小。Raha 会自动为这两种算法各自生成一系列参数。

2. 模式违规检测算法 [11] 从数据值与预定义数据模式是否兼容的角度评估数据值的正确性。为了不依赖用户提供的模式，我们基于字符袋表示 [20] 生成多种字符检查器。

3. 规则违规检测算法 [6] 基于数据值是否符合完整性约束来评估其正确性。我们纳入检查列间依赖关系的规则违规检测策略，例如函数依赖。单列规则，例如取值范围和长度，则已由离群点检测和模式违规检测算法隐式覆盖。Raha 默认认为每一列都依赖于其他所有列，并把这些函数依赖中的每一个都作为基础检测器。

4. 知识库违规检测算法 [4] 通过将数据值与知识库中的数据进行交叉检查来评估其正确性，例如使用 DBpedia [2]。我们检查当前数据集中数据值与 DBpedia 内所有实体关系的一致性。

检测器数量通常取决于列数、数据集中的字符分布以及知识库规模。在实践中，Raha 会通过为每个错误检测算法配置广泛的参数集合，生成数千个基础错误检测器。例如，对于 Flights 数据集，会生成 4174 个检测器。这种方法让用户无需选择和配置检测算法。

Raha 收集这些检测器的输出，以便为每个数据单元构造特征。每个数据单元的特征向量表示哪些检测器把该特定数据单元标记为数据错误。数据单元 `d[i, j]` 的特征向量，是所有错误检测策略 `s ∈ S` 在该数据单元上的输出组成的向量。形式化表示为：

```text
v(d[i, j]) = [s(d[i, j]) | ∀s ∈ S]              (1)
```

其中：

```text
s(d[i, j]) = 1，当且仅当 s 将 d[i, j] 标记为数据错误；
s(d[i, j]) = 0，其他情况。
```

因此，具有相似数据质量问题的数据单元会拥有相似的特征向量，因为它们会被相同的检测器标记。

接下来，Raha 遵循基于聚类的采样与标注流程，为每个数据列训练一个分类器。它基于特征向量相似性对每一列中的数据单元进行聚类。Raha 采样一组元组，使其覆盖尽可能多的未标注聚类，并要求用户将这些采样元组中的数据单元标注为脏或干净。

形式化地，Raha 在每次迭代中基于如下 softmax 概率函数抽取一个元组 `t*`：

```text
P(t) = exp(Σ(g∈t) exp(-Ng)) / Σ(t'∈d) exp(Σ(g∈t') exp(-Ng))              (2)
```

其中，`Ng` 是数据单元 `g` 当前所在聚类中已标注数据单元的数量，`exp` 是以 `e` 为底的指数函数。该评分公式会偏向那些其数据单元大多属于较少获得标签的聚类的元组。Raha 会将每个用户标签传播到同一聚类中的所有值，以增加有标签训练数据点的数量。

最后，Raha 为数据集中的每一列训练一个分类器，用于预测列内每个值的最终标签。Raha 的结果是一个与输入数据集维度相同的布尔矩阵，其中包含原始数据集中每个数据值的预测结果 `1` 或 `0`，即错误或干净。

### 2.2 错误纠正引擎

Baran 的任务是修复先前检测到的数据错误。与 Raha 类似，Baran 依赖一组基础错误纠正模型 `M = {m1, m2, ..., m|M|}`。Baran 基于可用的数据错误上下文生成纠错模型。对于每个数据错误，通常有三类上下文信息可用。数据错误上下文的一部分可以是值本身，也可以是同一元组中共现值所定义的邻近上下文，还可以是同一列中所有值所代表的领域。因此，我们有三类纠错模型。

1. 基于值的错误纠正模型只使用错误值本身的字符来学习如何修复数据错误 `e` [9]。基于值的模型在两个粒度级别上学习值转换：（1）精确匹配；（2）模式匹配。在精确匹配中，基于值的错误纠正模型会根据错误值的确切取值学习转换，例如把 `Holland` 替换为 `Netherlands`。在模式匹配中，基于值的模型会根据错误值的 Unicode 模式学习转换。例如，对于所有编码为 `<Nd><Nd><Po><Nd><Nd><Po><Nd><Nd><Nd><Nd>` 的日期，将 `/` 替换为 `.`，也就是把 `16/11/1990` 替换为 `16.11.1990`。

2. 基于邻近上下文的错误纠正模型基于列关系学习修复数据错误。基于邻近上下文的模型会根据其他列中干净数据值之间的关系，把活动领域中的干净值作为潜在纠正提出。我们把每一个一属性到一属性的函数依赖都视为一个基于邻近上下文的模型，用于根据左侧值生成纠正候选。

3. 基于领域的错误纠正模型使用错误所在列中的已有值来学习修复数据错误。基于领域的模型会从活动领域中提出最相关的干净值作为潜在纠正。它认为一个数据列中出现频率更高的干净值，更可能成为同一列中某个数据错误的纠正值。

每个错误纠正模型 `m` 会为每个数据错误 `e` 提出一个潜在纠正 `c`。Baran 会为每一对数据错误与纠正候选 `(e, c)` 生成一个特征向量。该特征向量的每个分量，对应于某个错误纠正模型 `m` 对用纠正候选 `c` 替换数据错误 `e` 的置信度。形式化表示为：

```text
v(e, c) = [P(c|em) | ∀m ∈ M]              (3)
```

其中，`M` 是所有错误纠正模型的集合，`P(c|em)` 是模型 `m` 向数据错误 `e` 提出纠正候选 `c` 的置信度。

当一个特征向量大多包含接近 1 的概率时，即对大多数模型 `m ∈ M` 而言 `P(c|em) ≈ 1.0`，纠正候选 `c` 更可能是数据错误 `e` 的真实纠正。原因是，在这种情况下，大多数高置信度错误纠正模型都为该数据错误提出了这一纠正候选。

随后，Baran 使用人工监督获得少量纠错示例。虽然 Baran 有自己的采样技术来选择待标注元组，但在端到端流水线中，它只是使用检测阶段已经标注过的同一批元组。Baran 会使用来自用户的这些新纠错示例，增量更新错误纠正模型。与 Raha 的检测步骤类似，Baran 会为每个数据列训练一个分类器，用于从所有提出的潜在纠正中预测每个数据错误的真实纠正。

### 2.3 从先前已清洗数据集中学习

Raha 和 Baran 这两个清洗引擎都能从先前已清洗数据集的知识中受益，但二者受益方式不同。

Raha 在内部运行数千个检测器，因此初始特征化阶段的运行时间较高。我们这样设计 Raha，是为了以运行时间为代价让系统完全无配置。我们不希望在检测器选择过程中依赖用户，从而让系统也能被更广泛的用户群体使用。自然地，相比其他检测器，某些检测器会为 Raha 的聚类算法创建更好的特征。尤其是，有些检测器可能只是生成噪声特征。移除这些检测器将有利于运行时间，也可能提升检测过程的有效性。

由于我们事先不知道哪些检测器最不有用，同时也不希望依赖用户来做出这个并不简单的决定，因此我们选择把它们纳入进来，并希望高特征维度能够抵消它们对整体检测性能的影响。不过，可以根据检测器在先前清洗任务中的相关性过滤掉其中一部分。在 Raha 中，我们会将当前列与先前已清洗数据集中的列进行逐列相似性检查。相似性基于可自动抽取的特征，这些特征描述列的内容和结构。

相似性计算完成后，会为每个策略生成一个得分，该得分结合了列相似性以及特定检测器在历史数据集上的 F1 分数。随后选择得分最高的一组检测器。我们在 Raha 论文中报告的实验表明，减少检测器集合可以显著降低特征生成运行时间，同时不会损害聚类有效性 [15]。

Baran 引擎以另一种方式从先前已清洗数据集中受益。具体来说，Baran 会收集过去观察到的所有清洗后值，用于训练基于值的纠错器。通过这种方式，Baran 不仅有机会获得数据集外的纠正候选，还会收集纠错证据，帮助预测模型更快收敛以选择正确纠正。在我们的原型中，我们从 Wikipedia 修订历史的更新中收集纠正，这会产生数百万个纠正候选，其中一部分会与当前数据集相关。

### 2.4 打包结构

我们已经在 GitHub 和 Python Package Index 上发布了两个系统的代码。我们的 Python 包包含三个独立类，分别面向数据集、错误检测和错误纠正。数据集类具有加载、规范化、比较、编辑和保存数据帧的方法。数据集类也具有基于数据清洗结果创建和评估已清洗数据帧的方法。错误检测类和错误纠正类覆盖 Raha 与 Baran 的功能，并实现了以数据集对象作为输入的方法。通过这些方法，用户可以初始化数据集、运行基础错误检测器或纠错器、生成特征、采样元组、训练分类器，并应用分类器预测数据错误或纠正。

图 2：使用我们的 Python 包构建的错误检测流水线示例。

图 2 展示了一个基于我们 Python 包构建的错误检测流水线示例。首先导入 `raha` 模块。接着，实例化 `Detection()` 类，并把 `app` 对象的 `LABELING_BUDGET` 属性设置为 20 个元组。然后通过调用 `initialize_dataset(dataset_dictionary)` 方法创建数据集对象，该方法只需要脏数据集的名称和路径作为输入。

接下来，调用 `run_strategies(d)`、`generate_features(d)` 和 `build_clusters(d)` 方法，为数据集的数据单元构造特征并构建聚类模型。随后，通过调用 `sample_tuple(d)` 方法迭代采样一个元组，并要求用户通过自定义标注函数标注采样元组。在我们的演示中，我们展示了一个手动标注场景。然后，`propagate_labels(d)` 方法将用户标签传播到聚类中并训练分类器。最后，可以调用 `predict_labels(d)` 来预测数据集中每个属性值的正确性。

该包的模块化设计为用户提供了使用灵活性，因为用户可以通过调用不同函数轻松编排工作流的不同步骤。

在下一节中，我们将描述原型用户界面如何使用户能够检查流水线中的不同步骤和资源，例如值如何聚类、用户如何提供示例，以及迁移学习如何影响每个分类任务的特征。

## 4. 相关工作

错误检测是检测错误数据值的任务 [1]。既有工作利用多种技术和资源来检测数据错误，例如数据增强 [10]、Web 表格 [22]、知识库 [5]、否定约束 [8] 和集成方法 [1, 21, 13, 16]。错误纠正，也称为数据修复，是修复已检测数据错误的任务 [19]。相关工作利用了多种信号，例如完整性规则 [3, 8]、外部来源 [5]、主动学习 [12, 24]、统计似然 [23]，或规则与统计的组合 [19]。

上述一些方法也曾被演示过 [24, 8, 5, 12, 25]。在本文中，我们提出演示一条端到端流水线，其中承载了我们最近发表的系统 Raha [15] 和 Baran [14]。具体来说，我们给出了对一个由示例驱动的整体框架的新洞见，该框架能够纳入独立的错误检测器和纠错器，并利用迁移学习。虽然先前演示展示了独立系统的内部机制，但我们的演示提供了对机器学习方法如何有效聚合并组合数千个检测器和纠错器的观察。

我们的界面支持深入查看每个基线检测器或纠错器的性能，以及它们如何被生成并用于创建特征向量。此外，我们还可视化了所生成特征向量的有效性，并展示迁移学习如何提高数据清洗流水线的效率和效果。

## 5. 结论

我们提出演示一条基于数据清洗系统 Raha 和 Baran 的端到端数据清洗流水线。我们的无配置系统只需可以忽略不计的数据标注工作量，就能达到较高的数据清洗性能。此外，我们的端到端数据清洗流水线还能从迁移学习中受益。
