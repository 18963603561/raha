# Baran 数据清洗

## 1. 目标

Baran 核心任务是已经知道哪些单元格有错以后，应该把这些错误值修成什么。

示例字段：

| 姓名  | 身份证                | 手机号            | 地址     |
| --- | ------------------ | -------------- | ------ |
| 张三  | 11010119900307653X | 13800138000    | 北京市东城区 |
| 张三  | 11010119900307653X | +8613800138000 | 北京市东城区 |
| 王五  | 310101198812216532 | 13900139000    | 上海市黄浦区 |
| 赵六  | 110101199009096531 | 1380013A000    | 上海市东城区 |
| 钱七  | 440101199505086536 | 13800138000    | 广州市天河区 |

假设上游 RaHa 已经检测出以下错误位置：

| 行号  | 字段  | 错误值            | 期望修复方向           |
| --- | --- | -------------- | ---------------- |
| 2   | 手机号 | +8613800138000 | 去掉国家码，修成 11 位手机号 |
| 4   | 手机号 | 1380013A000    | 把字母修成正确数字        |
| 4   | 地址  | 上海市东城区         | 结合上下文修成更合理地址     |

Baran 关注的是纠正结果，例如：

| 错误值            | 修复值         |
| -------------- | ----------- |
| +8613800138000 | 13800138000 |
| 1380013A000    | 13800138000 |
| 上海市东城区         | 北京市东城区      |

## 2. Baran 是什么

Baran 是让多个错误纠正模型从不同上下文提出候选修复值，然后用少量用户标注修复示例训练分类器，在候选值中选择最可信的最终修复。

## 3. Baran 完整纠正流程

Baran 的完整流程可以概括为：输入错误位置，采样用户修复，更新纠错模型，生成候选值，生成候选特征，训练分类器，预测最终修复值。

## 5. 错误纠正模型

Baran 使用三类基础错误纠正模型。每类模型关注一种上下文。

### 5.1 模型类型

| 错误定义 | 模型类型       | 主要看什么   | 能解决什么问题           | 示例                                |
| ---- | ---------- | ------- | ----------------- | --------------------------------- |
| 句法错误 | 基于值的模型     | 错误值本身   | 格式错误、拼写错误、常见字符串修复 | `+8613800138000` 修成 `13800138000` |
| 语义错误 | 基于邻近上下文的模型 | 同一行其它字段 | 列间关系错误、上下文不一致     | 同一身份证通常对应同一地址                     |
| 句法错误 | 基于领域的模型    | 同一列已有值  | 缺失值、低频错误、列内常见值替换  | 手机号列中常见 `13800138000`             |

这三类模型分别利用“值上下文、行上下文、列上下文”。一个错误值可以同时被多个模型提出修复候选。

### 5.2 模型参数配置

| 模型类型       | 算法或子模型 | 参数或配置           | 参数作用               | 常用配置示例                  |
| ---------- | ------ | --------------- | ------------------ | ----------------------- |
| 基于值的模型     | 错误值编码器 | 身份编码、Unicode 编码 | 控制模型看原始值，还是看字符类别模式 | `identity`、`unicode`    |
| 基于值的模型     | 纠正算子   | 删除、添加、替换、交换     | 控制如何从错误值生成候选修复值    | 删除 `+86`、替换 `A` 为 `8`   |
| 基于邻近上下文的模型 | 字段依赖关系 | 左侧字段、右侧字段       | 控制用哪一列推断另一列        | `身份证 -> 地址`、`姓名 -> 手机号` |
| 基于领域的模型    | 列领域分布  | 目标列             | 控制从哪一列已有干净值中选候选    | 手机号列、地址列                |

需要注意：Baran 的模型不直接保证候选一定正确。模型负责多提候选，后面的分类器负责综合判断哪个候选最可信。

## 6. 纠正模型举例

### 6.1 基于值的模型

基于值的模型只看错误值本身。它有两个维度：怎么看错误值，以及怎么修改错误值。

| 维度  | 可选项        | 含义                                   |
| --- | ---------- | ------------------------------------ |
| 编码器 | 身份编码       | 原样看错误值，例如直接看 `+8613800138000`        |
| 编码器 | Unicode 编码 | 看字符模式，例如数字、字母、符号                     |
| 算子  | 删除         | 删除多余片段，例如删除 `+86`                    |
| 算子  | 添加         | 添加缺失片段，例如把 `Str` 补成 `Street`         |
| 算子  | 替换         | 替换错误字符，例如把 `A` 替换为 `8`               |
| 算子  | 交换         | 整体替换，例如把 `Holland` 替换为 `Netherlands` |

这里的 `identity` 和 `unicode` 是两种“看错误值”的方式：

| 编码方式       | 中文理解 | 看什么               | 适合什么问题            | 示例                                  |
| ---------- | ---- | ----------------- | ----------------- | ----------------------------------- |
| `identity` | 身份编码 | 原始字符串本身           | 具体值修复、别名修复、固定写法修复 | 看到 `Holland`，学习整体替换为 `Netherlands`  |
| `unicode`  | 格式编码 | 字符类型模式，例如数字、字母、符号 | 格式修复、日期修复、手机号格式修复 | 看到 `16/11/1990` 的格式，学习把 `/` 替换为 `.` |

用手机号理解：

| 原始值            | `identity` 看到什么        | `unicode` 看到什么 |
| -------------- | ---------------------- | -------------- |
| +8613800138000 | 具体字符串 `+8613800138000` | 符号 + 一串数字      |
| 1380013A000    | 具体字符串 `1380013A000`    | 数字 + 字母 + 数字   |

```text
基于值模型数 = 2 个编码器 × 4 个算子 = 8 个模型
```

具体 8 个基于值子模型如下：

| 模型                   | 怎么看 | 怎么改  | 举例                           |
| -------------------- | --- | ---- | ---------------------------- |
| ① Identity + Remove  | 原样看 | 删除   | `U.S.` -> `US`               |
| ② Identity + Add     | 原样看 | 增加   | `US` -> `U.S.`               |
| ③ Identity + Replace | 原样看 | 替换   | `Bejing` -> `Beijing`        |
| ④ Identity + Swap    | 原样看 | 整体替换 | `Holland` -> `Netherlands`   |
| ⑤ Unicode + Remove   | 看格式 | 删除   | 身份证删除 `-`                    |
| ⑥ Unicode + Add      | 看格式 | 增加   | `Str` -> `Street`            |
| ⑦ Unicode + Replace  | 看格式 | 替换   | `16/11/1990` -> `16.11.1990` |
| ⑧ Unicode + Swap     | 看格式 | 整体替换 | 很少使用，适合模式相似但整体值要替换的场景        |

手机号示例：

| 错误值            | 候选修复值       | 使用的值模型          | 修复含义          |
| -------------- | ----------- | --------------- | ------------- |
| +8613800138000 | 13800138000 | Identity + 删除   | 删除前缀 `+86`    |
| 1380013A000    | 13800138000 | Identity + 替换   | 把 `A` 替换为 `8` |
| 1380013A000    | 13800130000 | Unicode 编码 + 删除 | 认为字母位置异常，删除字母 |

基于值模型适合处理格式错误和字符错误。

### 6.2 基于邻近上下文的模型

基于邻近上下文的模型看同一行其它字段。它的核心思想是：如果某个字段可以帮助推断另一个字段，就可以用这个关系生成候选修复值。

示例关系：

| 邻近字段 | 目标字段 | 模型含义                |
| ---- | ---- | ------------------- |
| 身份证  | 地址   | 同一身份证或身份证区划可能对应稳定地址 |
| 姓名   | 手机号  | 同一客户姓名可能对应稳定手机号     |
| 身份证  | 手机号  | 同一身份证可能对应稳定手机号      |

对 4 个字段：

```text
A = {姓名, 身份证, 手机号, 地址}
```

如果枚举所有字段对，则邻近上下文模型数量是：

```text
邻近上下文模型数 = 字段数 × (字段数 - 1) = 4 × 3 = 12
```

示例：

| 错误位置              | 同行上下文                    | 候选修复值       | 来自哪个模型    |
| ----------------- | ------------------------ | ----------- | --------- |
| 地址 = 上海市东城区       | 身份证 = 110101199009096531 | 北京市东城区      | 身份证 -> 地址 |
| 手机号 = 1380013A000 | 姓名 = 赵六                  | 13800138000 | 姓名 -> 手机号 |

基于邻近上下文模型适合处理语义错误，因为错误值本身可能看起来合法，但和同一行其它字段不一致。

### 6.3 基于领域的模型

基于领域的模型看同一列已有值。它会把该列中更常见、更像干净值的值作为候选。

以手机号列为例：

| 手机号            | 出现次数 | 是否适合作为候选  |
| -------------- | ---- | --------- |
| 13800138000    | 20   | 是         |
| 13900139000    | 15   | 是         |
| +8613800138000 | 1    | 否，已被检测为错误 |
| 1380013A000    | 1    | 否，已被检测为错误 |

领域模型可能给出：

| 错误值         | 候选修复值       | 理由         |
| ----------- | ----------- | ---------- |
| 1380013A000 | 13800138000 | 同列中常见且格式正常 |
| 1380013A000 | 13900139000 | 同列中存在且格式正常 |

每一列有一个领域模型。示例表有 4 个字段，因此：

```text
领域模型数 = 字段数 = 4
```

## 8. 用户标注与模型更新

Baran 会返回包含错误值的整行给用户，让用户直接给出正确值。

假设系统返回第 4 行：

| 姓名  | 身份证                | 手机号         | 地址     |
| --- | ------------------ | ----------- | ------ |
| 赵六  | 110101199009096531 | 1380013A000 | 上海市东城区 |

用户修复后：

| 字段  | 错误值         | 用户修复值       |
| --- | ----------- | ----------- |
| 手机号 | 1380013A000 | 13800138000 |
| 地址  | 上海市东城区      | 北京市东城区      |

这一个用户修复示例会同时更新多个模型：

| 更新对象      | 学到什么                      |
| --------- | ------------------------- |
| 基于值模型     | `A` 可以被替换为 `8`            |
| 基于邻近上下文模型 | 当前身份证上下文下，地址更可能是 `北京市东城区` |
| 基于领域模型    | 手机号列中 `13800138000` 是可信候选 |

Baran 的统一模型可以简单理解为：

```text
某个上下文下，某个候选值出现得越多，它越可信
```

论文中的表达是：

```text
P(c|em) = count(c|em) / count(em)
```

其中：

| 符号        | 含义           |
| --------- | ------------ |
| em        | 某个模型看到的错误上下文 |
| c         | 候选修复值        |
| count(em) | 这个上下文出现过多少次  |
| count(c   | em)          |

例如用户多次把 `1380013A000` 修成 `13800138000`，这个候选的概率就会提高。

## 9. 候选特征生成

Baran 不直接相信某一个模型，而是把所有模型对同一候选值的支持程度拼成向量。

以错误值 `1380013A000` 为例，候选值有：

```text
{13800138000, 13800130000, 13900139000}
```

假设只展示三个模型特征：

| 特征       | 含义           |
| -------- | ------------ |
| Value    | 基于值模型支持度     |
| Vicinity | 基于邻近上下文模型支持度 |
| Domain   | 基于领域模型支持度    |

候选特征如下：

| 错误值         | 候选修复值       | Value | Vicinity | Domain | 向量               |
| ----------- | ----------- | ----- | -------- | ------ | ---------------- |
| 1380013A000 | 13800138000 | 0.95  | 0.80     | 0.70   | [0.95,0.80,0.70] |
| 1380013A000 | 13800130000 | 0.40  | 0.00     | 0.10   | [0.40,0.00,0.10] |
| 1380013A000 | 13900139000 | 0.00  | 0.20     | 0.40   | [0.00,0.20,0.40] |

这张表就是分类器训练和预测的输入。

## 10. 训练模型并预测修复值

用户修复示例可以转成训练样本。

如果用户确认：

```text
1380013A000 -> 13800138000
```

那么训练样本可以写成：

| 错误值         | 候选修复值       | 向量               | 标签   |
| ----------- | ----------- | ---------------- | ---- |
| 1380013A000 | 13800138000 | [0.95,0.80,0.70] | 正确候选 |
| 1380013A000 | 13800130000 | [0.40,0.00,0.10] | 错误候选 |
| 1380013A000 | 13900139000 | [0.00,0.20,0.40] | 错误候选 |

分类器学到：多个模型同时支持的候选更可能是最终修复值。

| 第 4 行手机号 | 1380013A000    | 13800138000 | 选择    |
| 第 4 行地址  | 上海市东城区         | 北京市东城区      | 选择    |
